# Privacy & Trust — TrustShield

TrustShield is a **100% local-first** underwriting copilot. It is designed so a bank can run
fraud detection on sensitive loan packets **without any customer data ever leaving the machine**.
This document states exactly how that is enforced.

## 1. On-premise by construction — no external transmission

- **No runtime network calls.** Every service (forensics `:8001`, risk `:8002`, dashboard `:5173`)
  operates only on the local filesystem and talks only to `localhost`.
- **External registry checks are local mocks.** Verifications a production system would make against
  GSTIN / MCA21 / CERSAI / AIS / DigiLocker are served by adapters in [`shared/mocks/`](shared/mocks/)
  that read local JSON fixtures. The production interface (`ExternalVerificationAdapter._fetch`) is the
  single seam where a real HTTPS client would slot in; in this build it is never executed.
- **Automated enforcement.** [`scripts/verify_local_only.py`](scripts/verify_local_only.py) scans every
  source file and **fails the build** if any outbound-network usage appears outside the mocks/tests.
  It runs at the end of every phase.

The dashboard surfaces this guarantee to the investigator: *"All processing on-premise. No customer
data leaves this machine — every analysis is local."*

## 2. No raw PII in logs

The product rule is: **a score is never emitted without a human-readable evidence chain, and raw PII is
never written to a log.** Evidence items shown to the investigator in the UI legitimately contain
identifiers (that is the point of the tool), but anything written to a *log* is scrubbed.

[`shared/privacy.py`](shared/privacy.py) provides:

- **Pattern maskers** for the identifiers we can recognise without context:
  - **PAN** `ABMPS1234F` → `AB*******F` (first 2 + last 1 kept for audit correlation)
  - **Account / long numeric IDs** (9–18 digits) → all but the last 4 masked
  - **Property / survey IDs** `SY-911/2C` → `SY-***`
- **Field-aware dict redaction** (`redact_mapping`) that masks values under PII keys
  (`name`, `applicant_name`, `owner_name`, `pan`, `account_number`, `property_id`, `address`, …).
- **A logging filter** (`PIIRedactionFilter`) installed on the service and uvicorn loggers at startup
  (`install_log_redaction()`), so any record that does get logged is scrubbed as a safety net.

Income/loan amounts (≤ 8 digits) are intentionally **not** masked as account numbers, so operational
logs stay useful.

**Verification:** `tests/test_privacy.py` asserts that PAN, account numbers, and property IDs do not
survive in masked text, redacted dicts, or emitted log records.

## 3. Data handling & retention

- **All data in this repository is synthetic.** The packets under `data/synthetic/packets/` are
  generated by [`data/generator/`](data/generator/) with a fixed seed and contain **zero real PII** —
  fabricated names, PANs, and properties. They are committed on purpose so a fresh clone demos with no
  setup.
- **The cross-application graph is local runtime state.** It persists to
  `services/risk/graph_store/` (a pickle) and is **gitignored** — it is rebuilt deterministically from
  the packets. Nothing is uploaded.
- **The trained models** (`services/risk/models/`) are fit only on the synthetic set; the honest
  production answer is "retrain on the bank's own labelled history, on the bank's own hardware."

## 4. Auditability — every decision is explainable

TrustShield never returns a black-box score. Every `PacketDecision` carries:

- an **ordered, severity-tagged evidence chain** (forensic + semantic + model + graph findings), each a
  plain-English sentence with its source;
- the **learned model's contribution shown explicitly** (fraud probability + top feature attributions),
  never hidden inside the number;
- a **recommended action** (`approve` / `manual_review` / `freeze`) with its rationale and the exact
  thresholds and weights that produced it.

A hard action (**freeze**) is only issued when there is concrete document-level or cross-application
evidence — a model flag with no attributable evidence is routed to manual review, not auto-frozen. The
full report is exportable as JSON from the dashboard for an audit trail.

## 5. Honest limitations

- Name redaction in **free-text** logs is best-effort (names have no reliable pattern); the design
  relies on not logging name fields plus field-aware masking for structured data.
- The redaction patterns target Indian PAN / survey-number formats used in this build; a production
  deployment would extend them to the institution's full identifier set (Aadhaar, IFSC, CIF, etc.).
- This is a demonstration on synthetic data. It proves the pipeline and the privacy posture; it is not a
  certified compliance control.
